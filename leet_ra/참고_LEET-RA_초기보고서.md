# LEET-RA 시스템 개요 (개발자용)

LEET 추리논증 문항 출제를 위한 멀티 에이전트 파이프라인. v0.1 기초 스트럭처.

---

## 1. 전체 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│  터미널 1: 문항 추출 (기존 지문 → 문항)                      │
│                                                         │
│  [지문 입력] → analyze(A) → design(B) → review(C)        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  터미널 2: 지문 생성 (소재 → 지문 → 문항)                    │
│                                                         │
│  [소재] → architect(D1) → renderer(D2) → [터미널 1]       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  풀 파이프라인 (통합)                                      │
│                                                         │
│  [소재] → D1 → D2 → analyze → design → review            │
└─────────────────────────────────────────────────────────┘
```

---

## 2. 에이전트 5종 요약

| 에이전트 | 파일 | 역할 | 입력 | 출력 |
|---------|------|------|------|------|
| **A: analyze** | `leet_ra/prompts/analyze.md` | 지문의 논증 구조·추리 패턴 분석 | 지문 텍스트 | 논증 구조 맵 + 출제 포인트 + 오답 함정 후보 |
| **B: design** | `leet_ra/prompts/design.md` | analyze 결과로 문항 설계 | analyze 출력 | 발문 + 5지선다 + 해설 |
| **C: review** | `leet_ra/prompts/review.md` | 설계된 문항 독립 검수 | design 출력 | 8항목 검증 + 종합 판정 |
| **D1: architect** | `leet_ra/prompts/architect.md` | 소재 → 논증 구조 Blueprint 설계 | 소재 | Blueprint (매설 포인트 + 문단 배치도) |
| **D2: renderer** | `leet_ra/prompts/renderer.md` | Blueprint → LEET 문체 지문 렌더링 | D1 Blueprint | 지문 텍스트 (600~1000자) |

## 3. 에이전트 상세

### 3-1. Agent A (analyze) — 핵심 엔진

**목적**: 지문을 읽고 출제 가능한 모든 포인트를 구조화하여 추출.

**분석 4단계**:
1. **유형 판별** — 지문이 LEET 유형 체계(A~E) 중 어디에 해당하는지 분류
2. **구조 해부** — 유형별로 다른 분석 방식 적용:
   - A/C/E (논증 기반): 전제·결론·숨은 전제·반론 추출
   - B3 (논리 퍼즐): 요소·규칙·사례별 추론·확정 규칙표
   - D (법적 추론): 조문 정리·쟁점·대립 견해·미확정 변수
3. **출제 포인트 추출** — 유형·난이도·활용 발문까지 명시
4. **오답 함정 카탈로그** — T1~T10 코드로 매핑

**LEET 유형 체계 (5대 유형, 14소분류)**:

| 대유형 | 소분류 | 핵심 |
|--------|--------|------|
| A. 논증 분석·평가 | A1 전제·결론 / A2 강화·약화 / A3 오류 식별 | 숨은 전제, 논증 평가 |
| B. 연역 추리 | B1 명제 논리 / B2 양화 논리 / B3 논리 퍼즐 | 조건문, 규칙 복원, 배치 |
| C. 귀납·유비 추리 | C1 귀납 / C2 유비 / C3 인과 | 밀의 방법, 구조적 유사성 |
| D. 법적 추론 | D1 법 적용 / D2 판례 / D3 다견해 | 조문→사례, 해석 대립 |
| E. 논쟁 평가 | E1 다입장 강화·약화 / E2 입장 비교 | 3자 대립, 일치/차이 |

**오답 함정 코드표 (T1~T10)**:

| 코드 | 함정 | 예시 |
|------|------|------|
| T1 | 전제 역귀속 | 논증이 전제하는 것의 정반대를 전제로 제시 |
| T2 | 인과 방향 역전 | A→B를 B→A로 |
| T3 | 강화/약화 혼동 | 강화 요인을 약화로 |
| T4 | 범위 과대/과소 | 적용 범위 확장/축소 |
| T5 | 필요/충분 혼동 | 필요조건↔충분조건 |
| T6 | 비교 오류 | "같다"↔"적다/많다" |
| T7 | 주체/대상 혼동 | A 규칙을 B에 적용 |
| T8 | 변수 오산정 | 미확정 변수 값 오추정 |
| T9 | 범주 혼동 | 서로 다른 층위 동일시 |
| T10 | 부분 일치 오판 | 부분 일치→전면 일치 |

### 3-2. Agent B (design) — 문항 설계

**목적**: analyze 출력을 받아 5지선다 문항을 설계.

**설계 원칙**:
- 출제 포인트 **전량 소모** (analyze가 추출한 것 빠짐없이)
- 오답 매력도 전부 **중 이상** (자명한 오답 = 폐기)
- 오답 함정 **다변화** (같은 T코드 반복 금지)
- 설계 후 **역풀이로 정답 유일성** 반드시 확인

**선지 품질 게이트** — 아래 해당 시 폐기:
- 정답이 외부 지식으로만 판별 가능
- 오답이 상식으로 배제 가능
- 복수 정답 가능
- 발문-선지 논리 관계 불명확

**기출 발문 패턴 4종**: 법 적용("바르게 추론한 것"), 퍼즐("옳지 않은 것"), 견해 비교("평가로 옳은 것"), 논쟁("진술로 옳지 않은 것")

---

### 3-3. Agent C (review) — 독립 검수

**목적**: design과 **별도 세션**에서 문항을 검증. 편향 방지.

**검수 3단계**:
1. **직접 풀이** — 해설 보지 않고 지문만으로 직접 품
2. **8항목 검증**:
   - 정답 유일성, 복수 정답 위험, 오답 매력도, 발문 명확성
   - 지문 근거, 논리적 정합성, 난이도 적절성, 함정 다변화
3. **종합 판정** — ❌ 0개면 확정, 있으면 수정 후 재검수

**LEET 특유 출제 오류 체크**: 허위 딜레마, 조건 누락, 규칙 불완전 복원, 견해 과잉 해석, 강화/약화 오판, 비교 기준 불명확

---

### 3-4. Agent D1 (architect) — 지문 구조 설계

**목적**: 소재를 받아 출제 가능한 논증 구조 Blueprint를 설계. 텍스트 없이 뼈대만.

**설계 3단계**:
1. **D1-0 소재 스캔** — 논증 구조 내장 가능성, 추리 패턴 다양성, 반론·대립 구성 가능성, 조건·규칙 설정 가능성, 법적 추론 연결 가능성 5항목 평가
2. **D1-1 Blueprint 설계** — 핵심 주장/전제/숨은 전제/반론/재반박 + 매설 포인트(유형·난이도·발문) + 문단 배치도
3. **D1-2 자가 검증** — 출제 포인트 3개+, 난이도 혼합, 복수 유형 커버

**현재 상태**: v0.1 기초 구조. TODO: 유형별 소재 적합성 기준, 논증 구조 패턴 라이브러리, 매설 포인트 밀도 기준.

---

### 3-5. Agent D2 (renderer) — 지문 렌더링

**목적**: D1 Blueprint를 LEET 추리논증 문체로 실제 지문 텍스트 렌더링.

**렌더링 원칙**:
- 논증적 문체 (정보 전달 X, 논증 전개 O)
- 명료한 논리 구조 (전제→결론 흐름)
- 중립적 서술, LEET 수준 추상화
- 문단당 하나의 논점
- 숨은 전제는 명시하지 않되 추론 가능하도록 문맥 제공
- 분량: 600~1000자

**현재 상태**: v0.1 기초 구조. TODO: 기출 문체 분석·세칙, 금지 표현 목록, 정보 밀도 기준, Few-shot 예시.

## 4. 실행 방식 (OMC 위임)

모든 에이전트는 Claude Code의 **Agent 도구**로 서브에이전트를 spawn하여 실행한다.

### 위임 흐름

```
1. 사용자 트리거 감지 ("분석해줘", "문항 설계" 등)
2. 프롬프트 파일 Read (leet_ra/prompts/*.md)
3. Agent 도구 호출 (프롬프트 전문 + 사용자 입력을 prompt로 전달)
4. 결과 수신 → 사용자에게 전달
5. (파이프라인 모드) 이전 에이전트 출력을 다음 에이전트 prompt에 포함
```

### 라우팅 테이블

| 트리거 키워드 | 에이전트 | 모델 |
|-------------|---------|------|
| "분석해줘", "analyze" | ra-analyze | opus |
| "문항 설계", "design" | ra-design | opus |
| "검수", "review" | ra-review | opus |
| "지문 만들어줘", "generate" | ra-architect → ra-renderer | opus (순차) |
| "풀 파이프라인", "pipeline" | analyze → design → review | opus (순차) |

### 슬래시 커맨드

| 커맨드 | 동작 |
|--------|------|
| `/project:analyze` | 지문 → 구조 분석 |
| `/project:design` | 분석 결과 → 문항 설계 |
| `/project:review` | 문항 세트 검수 |
| `/project:generate` | 소재 → D1 → D2 → 지문 |
| `/project:pipeline` | 지문 → analyze → design → review |
