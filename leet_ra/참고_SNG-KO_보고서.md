# SNG-KO 시스템 보고서 -- 수능 국어 비문학 출제 파이프라인

> 작성일: 2026-04-14  
> 대상: 이어서 개발할 팀 개발자  
> 목적: 시스템 전체 이해 + 미완성 항목 파악 + 설계 원리 공유

---

## 1. 프로젝트 개요

### 1-1. 목표

수능 국어 비문학(독서) 영역의 지문과 문항을 LLM 멀티 에이전트 파이프라인으로 생성하는 시스템. 단순 텍스트 생성이 아니라, **변별 구조가 내장된 지문 설계 + 해당 구조를 소비하는 문항 세트 생성**이 핵심이다.

### 1-2. 핵심 차별점

- **공생성(Co-generation)**: 지문과 문항을 별도로 만들지 않는다. 지문 설계 단계에서 "이 문장이 어떤 문항으로 소비되는가"를 함께 설계한다.
- **생략 설계**: 변별의 핵심은 "지문에 쓰는 것"이 아니라 "지문에서 빼는 것"이다.
- **목표 난이도**: 수능 기출은 참고 방향이지 목표가 아니다. 기출 로직을 이해한 뒤 한 단계 이상 높인다.

### 1-3. 기술 스택

- **오케스트레이션**: Claude Code (OMC -- Open Multi-agent Collaboration)
- **에이전트 모델**: Claude Opus (핵심 에이전트), Claude Sonnet (문체 교정 D3, 미완성)
- **프롬프트 관리**: `sng_ko/prompts/` 디렉토리에 마크다운 파일로 관리
- **위임 방식**: Agent 도구를 통한 서브에이전트 spawn. 각 에이전트의 프롬프트 파일을 Read한 뒤 전문을 Agent의 prompt에 포함하여 위임.

## 2. 파이프라인 아키텍처

### 2-1. 전체 구조

```
[터미널 2: 지문 생성]
소재 원본
  |
  v
D1 (architect.md, opus) ---- 소재 스캔 + 변별 설계 + 자가 검증
  |                           Blueprint 출력 (텍스트 아님)
  v
D2 (renderer.md, opus) ----- KICE 문체로 지문 텍스트 렌더링
  |                           Blueprint를 충실히 반영
  v
D3 (d3_style.md, sonnet) --- 문체 교정 [미완성, 파일 미생성]
  |                           추론 불필요, 표면 문체만 교정
  v
지문 완성
  |
  +----- 터미널 1로 투입 -----+
                               |
[터미널 1: 문항 추출]          v
지문
  |
  v
scan (scan.md, opus) -------- 문장 단위 분해, 변별 요소 추출/등급화
  |                            1급/2급/3급 분류, 코어 공리 추출
  v
drill (drill.md, opus) ------ 상방 뚫림 1급만 입력, 심화 드릴
  |                            5개 드릴 강제 실행, 구체적 추론 경로 전개
  v
design (design.md, opus) ---- 5문항 세트 뼈대 설계
  |                            왜곡 도구 적용, 매력도 게이트
  v
review (review.md, opus) ---- 독립 에이전트로 교차 검수
  |                            ❌가 0개일 때만 확정
  v
문항 세트 확정


[풀 파이프라인]
소재 → D1 → D2 → D3 → scan → drill → design → review
(slash command: /project:full-pipeline)
```

### 2-2. 에이전트 간 데이터 흐름

| 출발 | 도착 | 전달 데이터 |
|------|------|-----------|
| D1 | D2 | Blueprint (1급/2급 변별 요소, 문단별 배치도, 방어 장치, 돌출 문장, 윤활유 계획, 교차 연결 설계) |
| D2 | scan | 완성된 지문 텍스트 |
| scan | drill | 상방 뚫림 1급 변별 요소 + 코어 공리 + 원본 지문 |
| drill | design | scan 전체 출력 + drill 심화 결과 |
| design | review | 원본 지문 + scan 데이터 + 5문항 뼈대 |

### 2-3. 설계 결정 사항

| 결정 | 이유 |
|------|------|
| D1은 분할하지 않음 | 소재 스캔 + 변별 설계 + 검증이 하나의 컨텍스트에서 이루어져야 함. 분할 시 컨텍스트 손실 위험 |
| D3만 Sonnet | 문체 교정은 추론 불필요, 표면 작업만 수행. 비용 절감 가능 |
| review는 독립 에이전트 | design과 같은 세션에서 실행하면 편향 발생. 별도 spawn으로 편향 없는 검수 |
| scan + drill은 순차 실행 | scan 결과를 drill에 그대로 전달. 컨텍스트 공유 필수 |
| 모든 핵심 에이전트는 Opus | 변별 설계와 추출은 최상위 추론 능력 요구 |

## 3. 프롬프트 시스템 상세

### 3-1. 파일 구조

```
sng_ko/prompts/
├── architect.md          # D1 마스터 -- 소재 스캔 + 변별 설계 + 자가 검증 (v2.1)
├── renderer.md           # D2 -- 구조 렌더링, KICE 문체 집필 (v2.1)
├── d3_style.md           # D3 -- 문체 교정 [미완성, 파일 미생성]
├── scan.md               # Agent A -- 변별 요소 추출 (v8.0)
├── drill.md              # Agent A-2 -- 심화 드릴 (v7.7)
├── design.md             # Agent B -- 문항 뼈대 설계 (v8.0)
├── review.md             # Agent C -- 검수 (v7.5)
├── instruction.md        # 소재 선정 터미널
└── subjects/             # 소재별 보조 모듈
    └── 과학기술.md        # D1 보조 + 기출 분석 데이터 포함
```

### 3-2. 마스터 + 모듈 아키텍처

```
D1 = Read(architect.md) + Read(subjects/{분야}.md) → Agent(opus)
D2 = Read(renderer.md) → Agent(opus)
D3 = Read(d3_style.md) + Read(subjects/{분야}_style.md) → Agent(sonnet)
```

- 마스터 프롬프트는 그대로 유지
- 소재별 모듈이 뒤에 부착되는 구조
- 모듈은 정보를 **추가**하지 제거하지 않으므로 성능 저하 위험 없음

### 3-3. 각 에이전트 역할 상세

#### D1 -- architect.md (지문 논리망 설계기)

**역할**: 소재를 분석하여 변별 구조가 내장된 지문 설계도(Blueprint)를 출력. 텍스트를 쓰지 않고 뼈대만 출력.

**3단계 구조**:
1. **D1-0 (소재 스캔)**: 핵심 개념 추출, 변별 잠재력 탐지 (7가지 구조 유형 체크), 소재 한계 분석
2. **D1-1 (변별 매설 설계)**: 1급/2급 변별 요소 설계, 왜곡 도구 역방향 적용, 문단별 배치도, 교차 연결 설계, 윤활유 배치
3. **D1-2 (자가 검증)**: 14개 항목 체크리스트. 하나라도 실패하면 D1-1로 돌아가 보강

**핵심 도구**:
- 난이도 패턴 4종 (A: 산출식 역추론, B: 상반 효과, C: 교차 문단 추론, D: 정의 범위 트릭)
- 왜곡 도구 12종 (W2~W14) -- 각 왜곡이 작동하려면 지문에 무엇이 필요한지 역방향으로 설계

#### D2 -- renderer.md (지문 텍스트 집필기)

**역할**: D1의 Blueprint를 평가원(KICE) 문체로 실제 지문 텍스트로 렌더링.

**핵심 규칙**:
- KICE 문체 규칙 (절대 원칙 5종 + 표기 규칙 4종 + 금지 표현 목록)
- 렌더링 규칙 7종 (문단 구조, 논리 생략, 윤활유 도포, 방어 장치 삽입, 돌출 문장, 교차 분산, 표현 수준)
- DPO 기반 세칙 15종 (실전 대조에서 도출된 세부 규칙)
- 정보 밀도 필터: 유효 명제/총 글자수 x 1000 = 18~22 기준

#### scan -- scan.md (변별 요소 추출)

**역할**: 지문을 문장 단위로 분해하여 변별 요소를 추출하고 등급화.

**핵심 기능**:
- 누적 읽기 원칙 (N번 문장 분석 시 1~N-1 모든 명제 참조)
- 3등급 분류 (1급: 교차 복합 추론/산출식/생략된 논리/돌출, 2급: 체인 가능 명제, 3급: 용어 정의)
- 상방 판정 (뚫림/막힘) + 5개 검증 체크리스트
- 코어 공리 추출 (1~3개)
- v8.0 신규: 혼동 유발 명제 탐색, 오답 매력도 예측, 정의 역적용 후보 추출

#### drill -- drill.md (심화 드릴)

**역할**: scan이 "상방 뚫림"으로 태깅한 1급만 입력받아 구체적 추론 경로를 강제 전개.

**5개 강제 드릴**: 변수 조작, 교차 문단 결합, 정의 역추론, 시나리오 치환, 매력적 오답 설계. 최소 3개에서 구체적 결과 필수.

#### design -- design.md (문항 뼈대 설계)

**역할**: 5문항 세트 뼈대 설계. 왜곡 도구 16종(W1~W16) 사용.

**핵심 규칙**:
- 매력도 하 선지 출력 금지 (절대 규칙)
- 매력도 하 발생 시 상향 프로토콜 적용
- v8.0 신규: W15(유사 개념 치환), W16(적용 범위 오인)

#### review -- review.md (검수)

**역할**: 원본 지문 + scan 데이터 + 문항 뼈대를 교차 검증.

**검증 항목 6종**: 1급 전량 소모, 정보 전량 소진, 판례주의 위반, 방어 장치 연동, 왜곡 논리 검증, 역방향 검증(복수 정답 위험).

## 4. 핵심 설계 원리

### 4-1. 생략 설계 (변별의 핵심)

이 시스템의 가장 중요한 설계 원리.

**원리**: 변별의 본질은 "지문에 쓰는 것"이 아니라 "지문에서 빼는 것"이다.

**구조**:
```
지문에 명시: 전제 A, 전제 B (각각 별도 문장, 별도 문단 가능)
지문에서 생략: 전제 A + 전제 B → 결론 C
수험생에게 요구: 전제 A + 전제 B를 결합하여 결론 C를 도출하는 추론
```

**검증**: 결론 C가 지문에 직접 진술되어 있으면 생략 설계 실패.

**생략 금지 대상**:
- 고등학생이 배경지식 없이 도달 불가능한 전제
- 방어 장치 (한정/양보 표현)
- 전문 용어 정의

**생략 대상**:
- 두 전제 결합의 자명한 결론
- 상반 효과의 한쪽 방향
- 대비되는 두 입장의 차이를 직접 정리하는 문장
- 특정 해석의 함의를 요약하는 문장

**기출 실증**: 정답 선지에 생략된 논리 추론이 집중 배치됨. "결론만 진술하고 이유를 생략 → 이유를 묻는 문항"이 평가원 실전 패턴.

### 4-2. 교차 연결 설계

```
| 전제 1 (문단 X) | 전제 2 (문단 Y) | 도출 결론 | 지문 명시 여부 |
|-----------------|-----------------|----------|--------------|
| (~)             | (~)             | (~)      | 명시하지 않음  |
```

- 최소 2개 필수
- 도출 결론이 지문 어디에서도 직접 진술되지 않아야 함
- D2(렌더러)가 이 결론을 렌더링하면 변별력 소멸

### 4-3. 논리 퍼즐 우선 원칙

- **금지**: 개념 A, B, C 병렬 제시 → 선지에서 조합 변경 (이것은 암기 테스트)
- **지향**: 인과 체인 안에서 중간 단계 생략, 조건 은닉, 방향 역전 → 논리 재구성 필요
- **자가 점검**: "수식을 외웠으면 풀리는가, 추론 경로를 따라가야 풀리는가?"

### 4-4. 정보 밀도 관리

| 지표 | 기준값 |
|------|--------|
| 정보 밀도 (유효 명제 / 총 글자 x 1000) | 18~22 |
| 문장당 평균 글자 수 | 45~60자 |
| 총 문장 수 | 25~30 |
| 윤활유 비율 | 15% 이내 |
| 유효 명제 수 | 28개 이상 |

### 4-5. 윤활유 설계

윤활유 = 1급 변별 요소 직전에 배치되는 긴장 완화용 문장.

**자격 기준**: 반드시 새 정보를 1개 이상 포함. 앞 내용 요약만 하는 문장은 "공기"이지 윤활유가 아님.

**금지**: 요약형 정리, 문단 예고, 메타 전환, 동어 반복

## 5. 과학기술 모듈 상세

### 5-1. 목적

`sng_ko/prompts/subjects/과학기술.md` -- D1 마스터에 부착하여 과학기술 소재의 변별 설계를 보조.

### 5-2. 난이도 강화 전략 6종

| ID | 전략 | 기출 수준 | 강화 방향 |
|----|------|----------|----------|
| A | 왜곡 복합화 | 단일 왜곡 | 왜곡 복합 적용, 다른 문단 조건과 결합해야 판별 |
| B | 생략된 전제의 역추론 | 정의 제시 → 적용 (1~2단계) | 한쪽 전제만 명시, 역(逆)에서 도출 + 제약 조건 결합 |
| C | 조건부 분기 심화 | "A이면 X, B이면 Y" → 오답 "A인데 Y" | 조건 경계 불분명, 조건 자체를 추론해야 함 |
| D | 미세 정의 구별의 시나리오 적용 | 정의 차이 아는 것만으로 풀림 | 미세 구별을 보기 시나리오에서 적용하여 귀결 추론 |
| E | 이유 생략 → 이유 추론 | 이유 생략 → 이유를 묻는 문항 | 이유 추론 시 다른 문단 전제와 결합 필요 |
| F | 경계 조건/한계 추론 | 경계 조건 직접 명시 | 경계 조건 미명시, 메커니즘 이해에서 실패 조건 도출 |

### 5-3. 인과 체인 최소 깊이: 4단계

기출 최고 난이도가 3단계이므로, 시스템 목표는 4단계 이상.

**체인 구조 3유형**:
- 직렬 체인: A → B → C → D (중간 단계 생략으로 변별)
- 분기 체인: A → B, A → C, B+C → D (상반 효과 확장)
- 교차 체인: A(문단1) + B(문단3) → C(생략) (교차 문단 추론)

### 5-4. 보기 설계 5유형

| 유형 | 핵심 | 기출 예시 |
|------|------|----------|
| 시나리오 적용형 | 복수 변수 동시 테스트 | 기출1 12번 (열팽창/바이메탈) |
| 추상 도식 대입형 | 조건별 다른 작동 구별 | 기출2 13번 (확산 모델) |
| 개념 전이형 | 분석 도구를 새 대상에 적용 | 기출4 17번 (L-그래프) |
| 역변환 추론형 | 결과물에서 변환 전 상태 역추론 | 기출5 16번 (360도 카메라) |
| 대조군 비교형 | 대조군 실패로 원본 강점 역추론 | 기출3 10번 (결측치/A기법) |

### 5-5. 보기 형식 구조

```
[대상 설정] → [변수/조건 설정] → [도식(선택)] → [관찰 결과] → [단서 조건(선택)]
```

### 5-6. 보기 설계 원칙

1. 코어 공리 전량 소진 (보기 1개로 공리 2~3개 동시 소모)
2. 결과 제시 → 원인 역추론
3. 오답은 단일 왜곡, 정답은 복합 추론
4. 변수 개수로 난이도 조절 (2개=중, 3개+=고)

## 9. 개발자 참고 사항

### 9-1. 품질 우선 원칙 (최상위 규칙)

- **최적화보다 품질이 항상 우선**
- 아키텍처 분할, 모델 교체 등 효율화 시 품질 저하 위험 있으면 반드시 경고
- 전례: D1 분할을 시도했다가 컨텍스트 손실 위험으로 취소

### 9-2. 에이전트 spawn 프로토콜

```
1. Read("sng_ko/prompts/{agent}.md") → prompt_text
2. (분야 모듈이 있으면) Read("sng_ko/prompts/subjects/{분야}.md") → module_text
3. Agent(
     name="kr-{agent}",
     model="opus",  # D3만 sonnet
     mode="auto",
     prompt="{prompt_text}\n\n{module_text}\n\n## 입력\n\n{사용자 입력}"
   )
4. 결과를 사용자에게 전달 (또는 다음 에이전트에 전달)
```

### 9-3. 운용 규칙

1. **scan 결과 확인 후 진행**: 1급 3개 이상, 윤활유 15% 이하, 코어 공리 추출 완료 확인
2. **design 후 review 필수**: review에서 결함(X) 0개일 때만 확정. 결함 있으면 루프
3. **보기 문항은 별도**: 현재 난이도 조절이 어려우므로 별도 작업
4. **소재별 성능 차이**: 과학 지문에서 가장 유의미한 결과. 다른 분야는 추가 튜닝 필요

### 9-4. 프롬프트 수정 시 주의사항

- architect.md와 renderer.md는 **생략 설계 원칙**으로 연동되어 있음. 한쪽만 수정하면 불일치 발생
- D1의 출력 형식(1급 요소의 명시/생략/추론 3분할)을 변경하면 D2의 렌더링 규칙 2-1도 함께 수정 필요
- scan.md의 등급 체계를 변경하면 drill, design, review 모두 영향
- 왜곡 도구(W1~W16)는 scan.md와 design.md 양쪽에서 참조. 추가/변경 시 양쪽 동기화
