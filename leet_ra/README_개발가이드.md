- 통합 테스트 완료 (5개 영역 × 1문항)

## 2. 파이프라인 구조

```
[sourcer] 소재 발굴 + 지문 초안 생성
    ↓ (사용자 승인 가능)
[analyze] 지문 분석 → 출제 포인트 추출
    ↓
[design] 발문 + 선지 ㄱㄴㄷ(또는 ①~⑤) 설계
    ↓
[review] 독립 세션에서 검수 (❌ 0개일 때만 확정)
```

## ★ 3. 핵심 원칙 (반드시 준수)

> 아래 원칙은 프로젝트 총괄자가 직접 지정한 것으로, 모든 개발 과정에서 최우선으로 준수해야 한다.

### 3-1. 지문 빌드 3대 원칙 (절대 규칙)

**원칙 1: 일반화**
- 고교 수준의 배경지식만 가진, IQ만 높은 사람도 **지문 자체는** 편하게 이해할 수 있어야 한다.
- 변별은 "지문 이해"가 아니라 "선지 판단"에서 발생한다.

**원칙 2: 실명/실제 학자명 금지**
- 마르크스, 칸트 등 실제 학자 이름은 **절대** 등장하지 않는다.
- '갑', '을', '병'으로 일반화하고, 이론도 `<이론>`으로 일반화하거나 '~라는 주장이 있다'는 식으로 일반화한다.

**원칙 3: 한국어 전용**
- 한국어가 아닌 언어를 사용하지 않는다. 특히 영어 이론 같은 건 등장하지 않는다.

### 3-2. 소재 발굴 대원칙

**임의 생성 허용**: 소재는 '환각'이 허용된다. 논리가 풍부하고 말만 되면 된다. 이유: 큰 수준의 일반화가 이루어지기 때문.

### 3-3. 영역별 절대 금지

| 영역 | 절대 금지 |
|------|---------|
| **법 (규범)** | **민법, 형법, 행정법 등 변호사시험 빈출 소재 절대 금지.** 특별법 영역(환경, 교육, 소비자 등) 사용. |
| **인문** | 기호 논리학(∀, ∃, → 등) 과도 사용 금지. '대우' 정도만 허용. 전공 개념 사용 시 지문 내 충분한 설명 필수. |
| **사회/과학** | 특별한 금지 없음. 소재 다양화에 초점. |

### 3-4. 선지 구성 핵심 규칙

- **Rule 4**: 매력도 하 선지 출력 금지 (절대 규칙)
- **Rule 16**: 지문 표현 2~3 어절 이상 동일 인용 금지. 반드시 패러프레이징. (규범 영역 예외 가능)
- **발문**: 기출의 전형적 발문을 그대로 사용. 변형 금지. (`sourcer.md` 세칙 5 참조)

### 3-5. 규정(R) 작성 세칙

- 시간 표현 명확히 (방향 + 기준 시점 명시)
- '아니한다' → '않는다' (과도한 법률 투 지양)
- '만 원' 띄어쓰기
- 사례에서 모든 전제 조건 명시 (거주 여부, 시점 등)

### 3-6. 기출 데이터 활용 시 가중치 원칙

**최근 3~5개년 기출이 90% 비중, 이전 기출은 10% 참고용.**

- 09~26학년도 전 연도 기출이 `/데이터/기출_정리_전체/`에 보관되어 있다.
- 전 연도를 파싱하여 W코드/유형을 확장할 수 있지만, **오래된 기출에서 발견된 새로운 W코드나 유형을 최근 기출과 동등한 비중으로 다루면 안 된다.**
- 이유: LEET 출제 경향이 해마다 변하며, 최근 기출이 현재 출제 방향을 가장 잘 반영한다.

## 5. 코드 체계 요약

### 5-1. 구성요소 코드 (Building Blocks)

지문은 아래 구성요소의 자유로운 조합으로 이루어진다.

| 코드 | 구성요소 | 세부 |
|------|---------|------|
| P | 줄글 | 연속 산문 |
| V1 | 견해 (독립) | 갑/을/병 각 1회 |
| V2 | 견해 (대화) | 갑-을-갑-을 교차 |
| R1~R4 | 규정 | 미번호/번호/번호+제목/비조문 |
| C | 사례 | 단일/복수 |
| PR | 원칙 | 해석 기준 |
| AS | 논증구조 | 원문자 문장 나열 |
| MR | 모형규칙 | 순수 추론 규칙 |
| MK | 마커 | ⓐⓑⓒ 참조 지점 |
| T | 표 | 실험/도식/대비/조건/수치 |

### 5-2. W코드 (오답 생성 원리) — 21개

상위 빈출: W1(결론 방향 역전, 10회), W12(계산 오류, 7회), W15(전문 지식 오류, 6회), W5(규칙 누락, 6회)

전체 목록 및 few-shot 예시: `기출_파싱_데이터/w_codes_merged.json`

### 5-3. AP코드 (매력도 패턴) — 11개

매력도 상: AP1(부분 진실), AP2(직관 부합), AP3(개념 근접), AP4(구조 모방), AP5(다단계 역전)

전체 목록: `기출_파싱_데이터/choice_logic_analysis.json`

### 5-4. 선지 구성 규칙 — 16개

핵심 규칙: Rule 4(매력도 하 금지), Rule 16(패러프레이징 필수)

전체 목록: `기출_파싱_데이터/choice_logic_analysis.json`의 `choice_construction_rules`

## 7. 향후 과제

### 7-1. 문항 품질 제고 (최우선)
- **지문 난이도 심화**: 현재 v1.5에서 정보 밀도, 추론 단계를 높여 실전 LEET 수준 도달
- **참신한 고난이도 문항**: 기출 패턴을 학습하되 새로운 논리 구조/소재 조합 발굴
- **DPO 세칙 구축**: 실전 문항 대조(좋은 예 vs 나쁜 예)를 통해 프롬프트 세칙 추가 (SNG-KO v8.0 수준 목표)

### 7-2. 소재 발굴 파이프라인 안정화
- **WebSearch 기반 법령/논쟁 자동 탐색** 품질 안정화
- **영역별 소재 풀 구축** (특히 법 영역: 특별법 데이터베이스)
- **소재 다양성 관리**: 세트 간 소재 중복 방지

### 7-3. 출력 파이프라인
- **한글(.hwp) 파일 출력**: 실제 LEET와 동일한 양식 (2단 구성)
- **출제 양식 자동화**: `/데이터/출제양식/` 템플릿과 연동

### 7-4. 운영 체계
- **1세트(40문항) 자동 생성 워크플로우** 안정화
- **검수 루프 자동화**: ❌ 발생 시 자동 수정 → 재검수
- **교열 매뉴얼 반영**: `/데이터/출제양식/` 내 교열 매뉴얼 규칙을 프롬프트에 통합

### 7-5. 데이터 확장 (선택 사항)
- 09~26 전 연도 기출 JSON 파싱 (현재 2개년 완료, 스키마 확립됨)
- W코드/유형 확장 가능. **단, §3-6의 가중치 원칙 엄수**: 최근 3~5개년 90%, 이전 10% 참고용
- 전 연도 해설 파싱 → W코드 매핑 확장

```
7. Agent(name="ra-design", model="opus", prompt="{prompt}\n\n## 입력\n\n{analyze 출력}")
8. Read("아키텍처/prompts/review.md") → prompt
9. Agent(name="ra-review", model="opus", prompt="{prompt}\n\n## 검수 대상\n\n{design 출력}")
```

### 8-2. 프롬프트 개선 사이클

```

4. 재생성 → 재피드백 → 반복
```
