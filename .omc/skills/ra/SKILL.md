# LEET-RA: 추리논증 출제 시스템

v1.5 기반 멀티에이전트 파이프라인.
소재 발굴 → 지문 생성 → 출제 포인트 분석 → 문항 설계 → 독립 검수를 자동 오케스트레이션한다.

## When to Activate

다음 트리거가 감지되면 이 스킬을 활성화한다:
- 사용자가 LEET/리트/추리논증/법학적성 관련 출제 작업을 요청할 때
- "지문 만들어줘", "문항 설계해줘", "검수해줘", "풀 파이프라인" 등의 자연어 명령
- sourcer, analyze, design, review 키워드
- 세트 단위 요청("법 3문항", "40문항 세트 만들어")

## Workflow

### Step 0: 모드 자동 판별

사용자 입력을 분석하여 모드를 결정한다:

| 입력 유형 | 판별 기준 | 모드 |
|----------|----------|------|
| 긴 텍스트 (지문 전문) | 500자 이상의 지문이 붙여넣기됨 | **분석 모드** (analyze → design → review) |
| 짧은 키워드/영역명 | "특별법 배타적 경제수역", "귀납 추리" 등 | **생성 모드** (sourcer → analyze → design → review) |
| 세트 요청 | "법 3문항", "40문항 1세트" | **세트 모드** (sourcer 반복 → … ) |
| 명시적 단계 지정 | "analyze만", "검수만" | **단일 모드** (해당 에이전트만 실행) |

### Step 1: 프롬프트 로드

실행할 에이전트의 프롬프트 파일을 Read한다:

| 에이전트 | 파일 경로 |
|---------|----------|
| sourcer (소재+지문 초안) | `leet_ra/prompts/sourcer.md` |
| analyze (Agent A) | `leet_ra/prompts/analyze.md` |
| design (Agent B) | `leet_ra/prompts/design.md` |
| review (Agent C) | `leet_ra/prompts/review.md` |

유형 체계 참조: `leet_ra/taxonomy/taxonomy.md`
W코드/선지 구성 규칙 참조: `leet_ra/data/w_codes_merged.json`, `leet_ra/data/choice_logic_analysis.json`

### Step 2: 에이전트 Spawn (OMC 오케스트레이션)

**각 에이전트는 반드시 Agent 도구로 opus 서브에이전트를 spawn하여 실행한다.**

```
Agent(
  name="ra-{에이전트명}",
  description="LEET 추리논증 {역할}",
  model="opus",
  mode="auto",
  prompt="{Read한 프롬프트 전문}\n\n---\n\n{입력 데이터}"
)
```

#### 생성 모드 파이프라인 (소재 → 지문 → 문항)

```
[1] ra-sourcer (Sourcer v1.6, opus)
    입력: 영역 지정(법/인문/사회/과학/논쟁) 또는 소재 키워드
    출력: 지문 초안 + 영역 태그 + 빌드 3대 원칙 준수 체크
    확인: 실명 0개, 한국어 전용, 일반화 통과
    ★ 사용자 승인 후 다음 단계로
         ↓
[2] ra-analyze (Agent A v1.5, opus)
    입력: 지문 초안(또는 기존 지문)
    출력: 구성요소 코드 + 출제 포인트 + W코드 매핑 + 논증 구조
    확인: 출제 포인트 3개+, 영역 확정, T코드/W코드 후보 추출
         ↓
[3] ra-design (Agent B v1.5, opus)
    입력: analyze 출력
    출력: 발문 + 5지선다(또는 ㄱㄴㄷ) + 해설
    확인: 매력도 하 = 0개, W코드 다변화, 정답 유일성 역풀이 통과
         ↓
[4] ra-review (Agent C v1.5, opus) ← 반드시 독립 에이전트로 분리
    입력: 원본 지문 + analyze 데이터 + design 문항
    출력: 검수 리포트(8항목)
    판정: ❌ 0개 → 확정 / ❌ 있음 → 해당 단계로 루프
```

#### 분석 모드 파이프라인 (기존 지문 → 문항)

sourcer를 건너뛰고 `[2] ra-analyze`부터 시작한다.

#### 세트 모드

영역별 문항 수를 사용자와 합의한 뒤, 영역별로 sourcer → analyze → design → review 파이프라인을 반복한다.
소재 중복 방지를 위해 이전 라운드의 소재 키워드를 다음 sourcer 프롬프트에 포함한다.

#### 단일 모드

사용자가 특정 단계만 요청하면 해당 에이전트만 spawn:
- "analyze만 해줘 + 지문" → ra-analyze만 실행
- "검수만 해줘" → ra-review만 실행 (이전 대화의 지문+analyze+design 참조)
- "소재 찾아줘" → ra-sourcer만 실행

### Step 3: 결과 전달 + 루프

- 각 에이전트 결과를 사용자에게 전달한다.
- review에서 ❌가 발견되면:
  - 정답 유일성/복수 정답 문제 → ra-design 재실행
  - 지문 근거 부족/오답 매력도 문제 → ra-analyze 재실행 또는 지문 보강
  - 출제 포인트 부족 → ra-sourcer로 돌아가 지문 자체를 보강
  - 재실행 후 ra-review 다시 spawn
- ❌ 0개가 될 때까지 루프한다.

## 핵심 원칙

1. **sourcer가 병목이다.** 좋은 소재 = 좋은 문항. 일반화·실명 금지·한국어 전용 3대 원칙 절대 준수.
2. **analyze가 엔진이다.** 출제 포인트 추출이 전체 품질을 결정한다. 반드시 opus.
3. **review는 독립 에이전트.** design과 같은 세션에서 절대 실행하지 않는다.
4. **모든 핵심 에이전트는 opus.** 추론 품질이 생명이다.
5. **기출 가중치 90/10 원칙.** 최근 3~5개년 기출이 90% 비중, 이전은 참고용 10%.

## 영역별 절대 금지 (프로젝트 총괄자 지침)

| 영역 | 절대 금지 |
|------|---------|
| **법 (규범)** | 민법·형법·행정법 등 변호사시험 빈출 소재 절대 금지. 특별법 영역(환경·교육·소비자 등) 사용. |
| **인문** | 기호 논리학(∀, ∃, → 등) 과도 사용 금지. '대우' 정도만 허용. 전공 개념 사용 시 지문 내 충분한 설명 필수. |
| **사회/과학** | 특별한 금지 없음. 소재 다양화에 초점. |

## Examples

```
# 영역 지정 생성 (생성 모드)
사용자: 특별법 영역으로 법 문제 1문항 만들어줘

# 세트 생성 (세트 모드)
사용자: 인문 5문항 세트 만들어줘

# 기존 지문 분석 (분석 모드)
사용자: 이 지문으로 문항 만들어줘
[지문 붙여넣기]

# 단일 에이전트
사용자: analyze만 돌려줘 [지문]
사용자: 검수해줘

# 풀 파이프라인
사용자: 풀 파이프라인 돌려줘 [소재 또는 지문]
```
